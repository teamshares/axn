# Changelog

## Unreleased
* [FEAT] Contract reflection (PRO-2842, groundwork for exposing one Axn to axn-mcp / axn-ruby_llm / a future axn-rest from a single definition): `MyAxn.input_schema` / `MyAxn.output_schema` return JSON Schema Hashes derived from `expects`/`exposes` (subfields, `model:`, `of:`, `shape:`, `inclusion:`, defaults, unions), and `Axn::Reflection::Values.serialize_exposed(result, configs)` renders a Result to a JSON-safe Hash. The public surface speaks **input/output** (the JSON Schema / OpenAPI / MCP `inputSchema` / LLM `parameters` lingua franca); the internal builder speaks the framework's own **inbound/outbound**. It's read-only and off the execution path, moved into core from axn-mcp, and `sensitive:` has zero effect on schema output. The schema faithfully mirrors runtime validation: a field is `required` unless it has a default or a validator that allows nil/blank ŌĆö a bare active validator (`numericality: true`) and a typed-but-no-presence field (`:boolean`/`:params`) are required, while `absence`/`acceptance` are nil-tolerant; a nil-allowed field folds `"null"` into its type/`anyOf` (and its `enum`); a union `type: [A, B]` becomes `anyOf`; `type: :params` maps to an object; presence-only and length-only fields are left untyped; a `model:` field's `<field>_id` is left unconstrained (it accepts any primary-key token) ŌĆö unless the caller already declared that same `<field>_id` as its own field, in which case the caller's declared property wins over the model-generated one and `required` is de-duplicated (order-independent ŌĆö whichever declaration comes first) rather than gaining a second entry ŌĆö the same holds for a nested `model:` subfield against an explicitly-declared nested `<field>_id` subfield on the same parent (only the explicit-id-first order is accepted at declaration; the reverse raises "duplicate sub-keys"); and reflected `default:`/`enum` values are deep-copied so a consumer can't mutate the runtime contract through the returned schema. Subfields (`on:`) nest under their parent (alias- and string-key aware), which is typed `object` and marked `required` unless a parent- or subfield-`default:` materializes it. **Known limitation:** only single-level subfield nesting is represented ŌĆö a dotted `on:` path or a subfield-of-a-subfield validates at runtime but is omitted from the schema, deferred until an adapter needs deep tool-input schemas (PRO-2844 / PRO-2845).
* [FEAT] Class-level `axn_name` and `description`, both inherited. `axn_name "ŌĆ”"` overrides the canonical display name resolved by `resolved_axn_name` (`axn_name` ŌåÆ `self.name` ŌåÆ `"Anonymous Axn"`), which is now the single source of an action's display name across **every** surface: the logging context stack, the `axn.call` / `axn.call_async` `ActiveSupport::Notifications` `resource` (the Datadog dimension), the `axn_stack` exception-report breadcrumb, and the vernier profiling label (filename-sanitized) ŌĆö each previously hardcoded its own `"AnonymousClass"` / `"Anonymous Class"` / `"AnonymousAction"` fallback, now unified to `"Anonymous Axn"`. A named class is unchanged; a class that sets `axn_name` reports under it everywhere, so a factory-built or adapter-wrapped tool gets one canonical identity. The async **enqueue/constantize** path is deliberately untouched ŌĆö it uses the real Ruby class name to re-materialize the worker, which `axn_name` must never shadow. `axn_name` is validated as a non-blank String at declaration (it feeds metric dimensions); it is display-only and never constantized. Class-level `description "ŌĆ”"` records a human-readable description (free-form, unvalidated), independent of the existing field-level `description:` metadata key.
* [FEAT] `semantic_hints :read_only, :idempotent, :destructive` ŌĆö a single, validated, **advisory** declaration of an action's side-effect / operational profile (nothing enforces it; the `_hints` suffix keeps that honest ŌĆö a `read_only` tool can still fire a destructive call, especially `idempotent`, which is undetectable). Inherited, and extensible: an adapter registers additional vocabulary (e.g. MCP's `:open_world`/`:closed_world`) via `Axn.extension_config.register_semantic_hint(ŌĆ”)`. Core stores and validates; adapters interpret (MCP tool annotations, REST default HTTP verb, RubyLLM safety gating).
* [FEAT] `ambient_context` ŌĆö a reserved, always-present parent on every Axn (its reader returns `{}` by default) for ambient caller identity (`current_user`, `company`, ŌĆ”), sourced differently per surface. Declare `expects :company, on: :ambient_context` and read `company` through the full validator stack; the parent is excluded from the input schema (so a client/LLM can never supply it) and reads are declaration-gated (a reader exists only for declared subfields). Per-invocation resolution: an explicit `ambient_context:` kwarg (adapters map their transport's context here; an explicit `nil` clears it) **replaces** the configured `Axn.config.ambient_context_provider`, else the provider result, else the default view over every registered `ActiveSupport::CurrentAttributes`, else `{}` ŌĆö then filtered to the declared ambient keys (a `model:` subfield's `<field>_id` is preserved), so the hash never carries a process-wide `Current` dump and undeclared attributes are neither readable nor injected. The provider is invoked at most once per call, and only when ambient subfields are declared; a provider failure surfaces as the action's own exception (memoized so it's never masked or double-raised, and never crashes exception-report building). `preprocess:`/`default:` are rejected on an ambient subfield (compute them in your provider or a `before` hook); `sensitive:` is supported. Migration payoff: replace a bare `Current.company` read with an `expects :company, on: :ambient_context` line ŌĆö validated, testable, callsite unchanged. Does **not** cross the `call_async`/Sidekiq boundary (an async-executed tool must pass explicit `ambient_context:`). Ships an opt-in `Axn/AmbientContextBypass` RuboCop cop (enable it like `Axn/UncheckedResult`) that flags direct `Current.<attr>` reads inside `include Axn` classes ŌĆö allowlisting `CurrentAttributes` lifecycle API and scoping to the innermost enclosing class ŌĆö steering toward a declared ambient subfield.
* [BREAKING] Exception reports (`on_exception` context) now carry a declared, sensitive-**filtered** `ambient_context` instead of the raw global `::Current.attributes`. Previously the report auto-attached the entire `::Current.attributes` **unfiltered** ŌĆö so any sensitive Current value leaked into the error tracker (`format_hash_values` did GlobalID/params coercion only, never key-based redaction). Now the framework attaches the resolved `ambient_context` (only the action's declared ambient keys) run through the same sensitive-field filter as `inputs`/`outputs` ŌĆö including dynamic `sensitive:` predicates (`sensitive: :method` / `-> { }`) ŌĆö under the `ambient_context` key; the reserved execution-context key `:current_attributes` is replaced by `:ambient_context`, and `ambient_context` becomes core's reserved subfield-parent name (the schema-exclusion default in the reflection builder moved from axn-mcp's `server_context` to `ambient_context`). Breaking for any error-tracker rule, `on_exception` handler, or spec keyed on the old `current_attributes` context key. Acceptable pre-1.0; the typed, filtered view strictly beats the old raw dump.
* [FEAT] Class-level extension-registry primitives so adapters (axn-mcp, axn-ruby_llm, future axn-rest) can hang transport-specific config off any Axn with no marker mixin or middle gem. `Axn.extension_config.register_semantic_hint(ŌĆ”)` extends the semantic-hint vocabulary (mirroring the existing `register_field_metadata_key`), and `Klass.set_extension_metadata(:adapter, **kwargs)` / `Klass.extension_metadata(:adapter)` give each Axn an inherited, copy-on-write per-adapter metadata bag ŌĆö so an adapter's `wrap(any_axn)` reads `klass.extension_metadata(:mcp)` directly. Hardening (Codex review): `extension_metadata(:adapter)` now returns a shallow `dup` of the stored Hash rather than the Hash itself, so a caller mutating the returned value (e.g. `klass.extension_metadata(:mcp)[:annotations] = ...`) can no longer write through into the inherited `class_attribute` bag in place and silently leak that mutation across parent/child/sibling classes ŌĆö bypassing `set_extension_metadata`'s copy-on-write reassignment. Further hardening (Codex review): `extension_metadata(:adapter)` now deep-copies nested `Hash`/`Array` structure, not just the top-level Hash, so mutating a nested value on the returned metadata (e.g. `klass.extension_metadata(:mcp)[:annotations][:extra] = 1`) can no longer leak into the stored metadata or into inherited subclasses either ŌĆö the earlier shallow `dup` protected only top-level keys. Deliberately not `Marshal`/ActiveSupport `deep_dup`: a metadata value can be a `Class` reference or a `Proc` (e.g. a `model:` klass), which must stay shared by identity rather than be cloned, so only `Hash`/`Array` containers are recursed and every leaf object is kept as-is. Still further hardening (Codex review): `extension_metadata(:adapter)` now also dups a mutable `String` leaf (e.g. `title: "T"`), not just `Hash`/`Array` containers ŌĆö so mutating a returned String in place (`klass.extension_metadata(:mcp)[:title].upcase!`) can no longer leak into the stored metadata either. A `Class`/`Proc`/`Symbol`/other leaf is still shared by identity, unchanged.
* [FEAT] Declared `tag`/`dimension` facets now surface as Sidekiq per-job `tags` at enqueue, so jobs are findable/filterable in the Sidekiq web UI (e.g. every job for a given company). Each facet becomes a `name:value` tag (array values fan out to one tag per element); a name declared as both a `tag` and a `dimension` yields two distinct tags. A new `Axn.config.sidekiq_job_tag_sources` (default `%i[tag dimension]`) selects which facet types participate ŌĆö set `%i[dimension]` for bounded-only, or `[]` to disable the sink. Only **input-phase** facets (`from: :inputs`) are surfaced, resolved from the **raw enqueued inputs** ŌĆö Sidekiq tags are an enqueue-time property (set before the job runs, in a different process), so `from: :result` facets are omitted, and `preprocess:`/`default:` are deliberately not applied (they run once at perform; applying them at enqueue would double-run side effects and could drift from the run). A `model:`-derived facet still triggers its lazy lookup if a resolver reads the record. Resolution is best-effort ŌĆö a failure never breaks the enqueue ŌĆö and computed tags union (deduped) with any static `sidekiq_options tags:` on the worker. Sidekiq-specific: ActiveJob has no native tag concept (its per-execution facets already ride the `axn.call` APM span). The knob is global for now; a per-action override is deferred (PRO-2856).
* [FEAT] Added `tag` and `dimension` class-level macros for declaring per-action observability facets. `tag` (high-cardinality) attaches to the `axn.call` OpenTelemetry span as `axn.tag.<name>` and to the notification payload as `tags:`. `dimension` (bounded) attaches as `axn.dimension.<name>`, as payload `dimensions:`, and is passed to `emit_metrics` via a new backward-compatible `dimensions:` keyword. Each `tag`/`dimension` declares a single facet (name + resolver, or name + block); the resolver runs from inputs before the action body by default (`from: :inputs`), or against the settled result when declared `from: :result`.
* [FEAT] Declared `tag`/`dimension` facets annotate `auto_log` output. When the configured logger is a `SemanticLogger` (e.g. via `rails_semantic_logger`), facets are forwarded to its tagged context as named tags (`axn.tag.<name>` / `axn.dimension.<name>`) so they become structured log fields (and dimensions are legible as Datadog log facets): input-phase facets tag every log line emitted during `call` plus the completion line, while `from: :result` facets tag only the completion line. With any other logger, facets are appended to the completion line as a readable `[tags: {ŌĆ”}] [dimensions: {ŌĆ”}]` suffix (respecting the existing context-truncation limit). Axn takes no dependency on `semantic_logger` ŌĆö it forwards only when the configured logger is already one.
* [FEAT] Resolved `tag`/`dimension` facets now attach to the global exception report: `Axn.config.on_exception` receives them as `context[:tags]` and `context[:dimensions]` (each present only when the action declares facets). A consumer handler that forwards `context` to its error tracker picks them up automatically; route `context[:dimensions]` to indexed/bounded tags and `context[:tags]` to freeform extra. Facet values are the same coerced, per-consumer-duped maps the span and `emit_metrics` see (a reporter callback mutating them can't corrupt other sinks). `:tags`/`:dimensions` are now reserved execution-context keys ŌĆö **breaking** only for code that set a `tags`/`dimensions` key via `set_execution_context` or the `additional_execution_context` hook (now stripped, framework facets win). Async exhausted/discarded-job reports (Sidekiq death handler / ActiveJob `after_discard`, i.e. the `:first_and_exhausted` / `:only_exhausted` modes) also carry the facets. Since no live instance survives, they're resolved best-effort against an action reconstructed from the job args ŌĆö deserialized (so GlobalID/model inputs restore) and run through the inbound preprocessing/defaults pipeline (so defaulted/preprocessed inputs match what the worker saw); input-derived facets resolve, output-derived ones are skipped per-facet.
* [FEAT] Add `Axn::Async.owns?(candidate)` ŌĆö a public predicate answering "is this job/notice signal Axn-owned?" for error-reporter `before_notify`-style filters that suppress backend-native duplicate reports (Axn already reports via `on_exception`). It accepts whatever an error reporter's Sidekiq/ActiveJob plugin hands you ŌĆö a resolved Class, a String class name (including the ActiveJob adapter's `"::ActiveJobProxy"` suffix), or a raw Sidekiq job Hash (string or symbol keys, checked `display_class` ŌåÆ `wrapped` ŌåÆ `class`) ŌĆö and folds in every detail of Axn's async wiring: `klass < Axn`, the generic Sidekiq worker (`klass <= Axn::Async::Adapters::Sidekiq::Worker`, covering both per-action `AxnSidekiqWorker` subclasses and the global `DefaultWorker`), the proxy naming convention, and the `display_class` wire format. Blank/unrecognized input returns `false` without raising. A new `Axn::Async.register_ownership_predicate { |klass| ... }` lets a future async adapter extend wrapper detection without any downstream filter change (the Sidekiq adapter registers its own). **This also fixes stale, currently-broken guidance:** the `docs/recipes/suppressing-duplicate-async-reports.md` recipe still showed `klass&.include?(Axn::Core)` (and a "Direct Sidekiq worker: component is the Axn action class itself" note), which silently stopped detecting Axn ownership once actions became generic-worker-wrapped rather than `Sidekiq::Job`s ŌĆö anyone following it had a live duplicate-reporting bug. The recipe now uses `Axn::Async.owns?`.
* [FEAT] `Axn::Configurable.overrides` now also generates `raw_<name>` for each `overridable:` setting ŌĆö the nearest class-level override in the ancestry, or `Axn::Configurable::UNSET` if none is set, with no config fallback and no `Setting#resolve` (unlike `resolved_<name>`, which always resolves and falls back to the library config). For a caller that needs to distinguish "no override" from "resolves to the library default" without reaching into the private `@_axn_config_overrides` ivar. All three generated accessors (`<name>`, `raw_<name>`, `resolved_<name>`) share a single internal ancestry walk via closure-captured helpers rather than dispatching through each other's public methods ŌĆö so a consumer action class that defines its own colliding `raw_<name>`/`resolved_<name>` class method can't shadow the resolution Axn's readers depend on.
* [BREAKING] Nested `call!` now aggregates base headers across every level. When an inner action's failure bubbles through ancestor `call!`s, each level's declared base `error` is prepended to the child's already-resolved presentation ŌĆö so `Outer.call.error` reads `"Outer: Inner: leaf"` instead of the previous `"Outer: leaf"` (the inner base was dropped). Each segment uses its own `join:`. A baseless ancestor passes the child's presentation through unchanged, and a parent's own matching reason (e.g. `error "Record not found", if: NotFoundError, standalone: true`) takes precedence over the bubbled child message. This composition is **bucket-independent** ŌĆö it applies to `fail!`, user-facing validation failures, `fails_on`-classified exceptions, and *unexpected* exceptions alike (for an unexpected exception only the declared bases chain, since there is no authored leaf; the raw exception message never enters `result.error` and a baseless level contributes nothing). `Axn::Failure#message` (and a user-facing `Axn::ValidationError#message`) is stamped with the same fully-resolved string on the executor's way out, so rescuing from `call!` sees the identical presentation as `result.error`; **foreign** exceptions reclassified via `fails_on` keep their original technical `#message` (never rewritten), with the user-facing presentation living only on `result.error`. Adds `Axn.owns_failure_exception?` (predicate for framework-owned exceptions whose `#message` axn may stamp; covers `Axn::Failure` and user-facing `Axn::ValidationError`). Exception *reports* (the `on_exception` context, which fires only for unexpected exceptions ŌĆö not for `fail!`/`fails_on` failures) also gain an `axn_stack` breadcrumb ŌĆö the nested `call!` chain (outermost ŌåÆ innermost) ŌĆö so a bug report shows which path reached it (omitted for a single, non-nested action; each entry uses the action's `resolved_axn_name`, so anonymous/factory-built actions render as `"Anonymous Axn"` unless they set `axn_name`). **Breaking / migration:** two observable string shapes change for existing callers ŌĆö a nested `result.error` now includes every level's base (`"Outer: Inner: leaf"`, previously `"Outer: leaf"`), and a rescued `Axn::Failure#message` (and user-facing `Axn::ValidationError#message`) now carries the resolved presentation rather than the raw `fail!` reason. To migrate: for display, read `result.error`; to branch on a specific failure, match the exception type or `result.exception` rather than the message string; update any tests, error-tracker rules, or callback/`rescue` matchers that pin these strings. Foreign (`fails_on`) exceptions are unaffected ŌĆö their `#message` is unchanged.
* [BREAKING] Removed the experimental `_include_retry_command_in_exceptions` config setting and the `retry_command` key it added to exception context. The feature embedded every field value into a single `Action.call(...)` string, which defeated key-based redaction in error trackers (Honeybadger `filter_keys` / Rails `config.filter_parameters` only scrub by key, so sensitive values the action hadn't marked `sensitive:` leaked inside the string even when the same fields were `[FILTERED]` in the Params panel). It was also anti-correlated with its own usefulness ŌĆö a runnable command only materialized for the simplest actions (scalar inputs, no `sensitive:` fields, no ActiveRecord/object dependencies), exactly the actions trivial to retry by hand from the structured `inputs` context, while degrading to `Model.find(id)` / `inspect` / `[FILTERED]` for everything else. The structured `inputs`/`outputs` context (with its per-key filtering intact) remains the supported way to inspect a failed call. Breaking only for code that set the flag ŌĆö experimental, underscore-prefixed, and `default: false`. The unrelated unpersisted-AR formatter fix (renders `#<ClassName (unpersisted)>` in `inputs`/`outputs`) is unaffected.
* [FEAT] Ship a hand-authored, agent-facing usage guide at the gem root (`AGENTS-consuming.md` ŌĆö the consuming-audience sibling of the repo's own `AGENTS.md`), packaged in the gem (it lands in `spec.files` via the existing `git ls-files` allowlist ŌĆö no gemspec change) so a consuming project's LLM can read it offline via `bundle show axn`. It's a dense cheat-sheet for code that *uses* Axn (not internals): the `expects`/`exposes`/`call` contract surface and options, result/invocation semantics (`call` vs `call!`/`call_async`), the `model:`/`on:` field resolvers, the failure-bucket model (`fail!` vs `fails_on` vs unhandled exception, `user_facing:`, base/reason message composition), the subtle gotchas (top-level-only indifferent access, `expects`+`exposes` auto-copy, `done!` rolling back a manual transaction, default `call`), and the strategies ŌĆö pointing out to the docs site and `lib/` source rather than inlining depth. README gains an "Using Axn with an AI agent" section with a copy-paste `AGENTS.md`/`CLAUDE.md` snippet. Deliberately *not* auto-generated from the VitePress site (`vitepress-plugin-llms`): the plugin only mechanically reformats the human docs (`llms-full.txt` is the whole site inlined ŌĆö verbose, low signal, crowds out reading the source), whereas the win here is hand-curation and density; the optional generated `llms.txt` was skipped as low-value and not near-zero-effort (the plugin isn't a dependency). A hosted-URL copy and a `rails g` rules generator are deferred while Axn is alpha (internal users only).
* [BREAKING] Consolidated automatic-logging controls: `log_calls` and `log_errors` are **removed** and replaced by a single declarative `auto_log`. `auto_log` resolves a level (or off) per outcome ŌĆö `success`, `failure`, `exception` (the values `result.outcome` reports). A positional level is the default for any outcome not named; `success:`/`failure:`/`exception:` keywords override it (each accepting a level, `false`, or `true` for the configured default); a bare `auto_log` / `auto_log true` / no declaration all log every outcome at `Axn.config.log_level` (unchanged default). With keywords but no positional, unnamed outcomes are off ŌĆö so `auto_log exception: :error` logs *only* raised exceptions, distinguishing an unhandled raise from an expected `fail!` (the two previously shared `log_errors`'s single "errors" bucket). The "About to execute" before line is emitted only when success logging is on, at the success level (so an errors-only config stays quiet until something fails), and an invalid level raises `ArgumentError` at declaration. Migration: `log_calls <lvl>` ŌåÆ `auto_log <lvl>`; `log_calls false` ŌåÆ `auto_log false`; `log_calls false` + `log_errors <lvl>` ŌåÆ `auto_log <lvl>, success: false`. `Axn::Factory.build` likewise replaces its `log_calls:`/`log_errors:` keywords with a single `auto_log:` (a level/bool, or a Hash of per-outcome levels). Fixes the prior wart where setting both `log_calls` and `log_errors` silently ignored the latter.
* [BREAKING] Actions can no longer be instantiated directly ŌĆö `new` is now a private class method. The sanctioned entry points (`.call` / `.call!`) build the instance internally, so `MyAction.new(...)` now raises `NoMethodError: private method 'new' called`. This closes an unsanctioned path that bypassed hooks, inbound/outbound validation, and the rest of the per-call guarantees (callers reaching for `.new` got a half-initialized instance and were confused when none of those ran). Use `.call` / `.call!`; the rare test that needs a bare instance can use `.send(:new, ...)`. Subclasses (and the now-no-op re-inclusion case below) inherit the private `new` automatically.
* [BUGFIX] Re-including `Axn` in a subclass of an existing action (`class Child < Parent; include Axn; end`) is now a no-op instead of silently wiping the parent's inherited `expects`/`exposes`. The redundant `include` previously re-ran setup, which re-declared the inheritance-aware `class_attribute`s that hold field configs and reset them to empty ŌĆö so `Child` lost the parent's entire contract and every input failed as unexpected. Inheritance already carries the configs (and everything else) down, so a second `include Axn` now returns early.
* [BREAKING] Field keys are now normalized to symbols, giving the contract indifferent access. Previously axn matched a declared field's key type *exactly* against the call-arg's key type, so `expects :note` called as `.call("note" => "x")` (e.g. splatting JSON/HTTP params straight into `.call(**body)`) silently read `nil` and failed validation with the generic `"Something went wrong"`. Now the wire key at declaration (`expects "note"` ŌēĪ `expects :note`), the inbound key at call time (`.call("note" => ŌĆ”)` ŌēĪ `.call(note: ŌĆ”)`), and the outbound key on `expose` (`expose("saved", v)` / `expose("saved" => v)` / a string `expose_return_as` all resolve to the declared `:saved`) are symbolized ŌĆö so every declared-key ├Ś supplied-key combination resolves identically and the `**params` footgun is removed. Normalization is **top-level keys only**; field *values* (including nested hashes) are untouched, and symbol-*value* async behavior without ActiveJob is unchanged (still raises `Axn::Async::UnserializableArgument`). Breaking only for code that declared string field keys or relied on string-keyed direct calls failing ŌĆö both previously either failed silently or were non-idiomatic. Async round-trips are unaffected: both deserialize paths already re-symbolize keys before re-invoking.
* [BREAKING] The Sidekiq adapter no longer turns the action into a `Sidekiq::Job`. Previously `async :sidekiq` did `include Sidekiq::Job` on the action, so the action class itself was the worker. Now a generic `Axn::Async::Adapters::Sidekiq::Worker` runs every action by name (`perform(action_class_name, kwargs)` ŌåÆ `action_class_name.constantize.call(**kwargs)`), and `call_async` enqueues that worker. **The action class is therefore no longer a Sidekiq job** ŌĆö `MyAction.sidekiq_options`, `MyAction.perform_async`/`perform_in`/`perform_at`/`.set`, and Sidekiq's class-level testing helpers (`MyAction.jobs`/`drain`/`perform_one`) are gone from the action; use `call_async` (the supported API, unchanged). This is what lets the action keep a private `new` (Sidekiq instantiates the worker, never the action) and fixes the previously-broken "rely on `set_default_async(:sidekiq)` without declaring `async`, run in a fresh worker process" path (the worker only needs `.call`, so no lazy adapter reconfiguration is required in the worker). Per-action configuration is unchanged in spelling ŌĆö `async :sidekiq, queue: "x", retry: 3` and the `async :sidekiq do sidekiq_options(...); sidekiq_retry_in { ... }; end` block form both work, now applied to a per-action `MyAction::AxnSidekiqWorker < Worker` subclass (so the full Sidekiq class-level DSL remains available). The Sidekiq Web UI still shows the real action name (carried as `display_class`); the underlying job class is `MyAction::AxnSidekiqWorker` for explicitly-configured actions, or a dedicated `Worker` subclass carrying the global default's config (kwargs + block) for actions riding the global default, and the job args are `[action_class_name, serialized_kwargs]`. Verified on Sidekiq 7 and 8.
* [FEAT] Added `expose(result)`: forward a nested action result's declared exposures (the intersection with the current action's `exposes`) in one call. Failure-tolerant; raises `Axn::ContractViolation::NoMatchingExposures` when there is no field in common.
* [FEAT] Added `inputs`: a reader returning the action's resolved declared-inbound fields as a Hash, for splatting into nested calls (`Child.call(**inputs, role: ROLE)`).
* [BREAKING] `inputs` is now a reserved expectation/exposure field name. An action declaring `expects :inputs` / `exposes :inputs` will raise `Axn::ContractViolation::ReservedAttributeError`.
* [FEAT] Add `user_facing:` to `expects` to mark a field whose violation is the caller's fault, not a bug. By default a failed `expects` validation is dev-facing ŌĆö it settles in the **exception** bucket, pages the global `on_exception` handler, and `result.error` is the generic `"Something went wrong"`. `expects :note, user_facing: true` reclassifies a violation of that field into the **failure** bucket instead: `on_failure` fires, the global report is skipped, and the field's validation message (`"Note can't be blank"`) surfaces on `result.error`. The value rides `fail!`/`error`/`fails_on` muscle memory ŌĆö `true` (use the field's own message), a String (override it), a Symbol naming an action method, or a callable receiving the `InboundValidationError`; one that resolves blank falls back to the field's own validation message. The surfaced message is a failure **reason**, attached under a declared base `error` by default (standalone when no base is declared), exactly like a `fail!` message. The field stays **required** (unlike `optional: true`, which removes the check) ŌĆö it changes *who is blamed*, not *whether* it's validated. It's a per-field decision: in a mixed failure where a `user_facing:` field **and** a plain field are both invalid, the dev-facing one **dominates** (the call still pages), so a real contract bug is never masked behind a friendly message. The structured `InboundValidationError` is preserved on `result.exception` either way. **`user_facing:` is top-level only** ŌĆö it can't be declared on a subfield (`on:`), and it's rejected on a field that carries nested expectations (subfields via `on:`, or a shape block); those member/structural checks and model consistency are always dev-facing (support for `user_facing:` on structured/nested fields is deferred until a concrete need appears). Replaces the `optional: true` + manual `fail!` workaround for "required, but user-supplied" fields without reaching for the heavier `use :form`.
* [FEAT] Add `Axn::Configurable` ŌĆö a small DSL for declaring configuration so gems built on Axn (e.g. axn-mcp, axn-ruby_llm) stop hand-rolling their own config object, `configure` yielder, validation, and test reset. `extend Axn::Configurable` on a module gives `.config` / `.configure` / `.reset_config!` from `setting` declarations (with `default:`, `one_of:`, `validate:`, callable read-time resolution via `callable:`, and a `<name>?` predicate); `overridable: true` settings get per-class overrides on actions that `include <Namespace>.overrides` (resolving nearest-ancestor override ŌåÆ library default). `extend Axn::Configurable::Settings` declares the same validated settings as instance accessors on a plain class. See the "Configuration for Axn-based Gems" recipe. Axn's own `Axn::Configuration` now declares its simple settings (`log_level`, `emit_metrics`, `raise_piping_errors_in_dev`, `async_max_retries`, `additional_includes`) via this DSL; side-effecting/computed settings (`env`, `logger`, `on_exception`, the async setters, `async_exception_reporting`, `rails`) remain hand-written. No public API change to `Axn.config` / `Axn.configure`.
* [BREAKING] `on_success` now fires after the **enclosing** transaction commits (immediately when none is open) and is **skipped on rollback**, rather than always firing inline. Previously, an action nested inside another action's transaction ran its `on_success` before the outer transaction committed ŌĆö so the side effect (email, HTTP call, enqueue) fired even when the outer transaction later rolled back. Implemented via `ActiveRecord.after_all_transactions_commit` (requires ActiveRecord 7.2+; no-op/inline without ActiveRecord). Failure-path callbacks (`on_failure`/`on_error`/`on_exception`) are unaffected and still fire immediately. No opt-out flag. Caveat: deferral tracks **joinable** transactions only (the `transaction` strategy and ordinary `ActiveRecord::Base.transaction` blocks); an action run directly inside an explicitly non-joinable transaction (`joinable: false`) runs `on_success` immediately, since Rails hides such transactions from `after_all_transactions_commit`.
* [FEAT] Async argument serialization now uses `ActiveJob::Arguments` whenever ActiveJob is loaded ŌĆö for **all** adapters, including Sidekiq. Within a deployment every backend now round-trips the same rich type set losslessly (GlobalID models, `Date`, `Time`, `DateTime`, `TimeWithZone`, `Duration`, `BigDecimal`, `Symbol`, `Range`, nested symbol-keyed hashes), fixing a latent bug where a Sidekiq-enqueued `expects :at, type: Time` arrived on the worker as a `String` (validation then failed) while the same action worked under the ActiveJob adapter. The one remaining asymmetry is documented: a deployment **without** ActiveJob accepts only JSON-native + GlobalID-able args (it has no rich serializer to use). `enqueue_all` static args follow the same rules ŌĆö rich types round-trip on the ActiveJob path, JSON-native + GlobalID-able on the fallback (otherwise raising `UnserializableArgument`).
* [BUGFIX] Enqueuing an async action with an unserializable argument now raises a field-aware `Axn::Async::UnserializableArgument` (naming the field, its class, and the fix) at enqueue time ŌĆö on both the ActiveJob path (wrapping `ActiveJob::SerializationError`) and the no-ActiveJob fallback path ŌĆö instead of silently corrupting it (Sidekiq would JSON-stringify a `Symbol`/`Date`/`Time`, or dump a `Tempfile`/custom object into the payload).

The no-ActiveJob fallback wire format (`_as_global_id` suffix for GlobalID args) is unchanged. Separately, the fallback path now **raises** on `Symbol`/`Date`/`Time`/`BigDecimal`/files/custom objects that previously passed through (lossily); pass JSON-native or GlobalID-able values, or load ActiveJob for the richer set.
* [FEAT] `model:` `expects` fields now also define a `<field>_id` reader whose single meaning is the record's primary key ŌĆö whether the action was called with `user:` or `user_id:`, `user_id` returns the pk. It triggers no extra lookup: for the default `:find` finder a supplied id is returned as-is; otherwise it reads the (memoized) resolved record's `.id`, so it's meaningful even with a custom finder (where the `<field>_id` key holds a lookup token, `user_id` still returns the actual pk). Alias-aware (`as: :raw_user` ŌåÆ `raw_user_id`) and silently defers (debug-logged) to any same-named method already declared. Composite primary keys aren't supported by the singular convention.
* [BREAKING] For the default `:find` finder, passing **both** a `model:` record and a contradictory `<field>_id` (e.g. `user: <id=5>, user_id: 9`) now raises `InboundValidationError` instead of silently preferring the record. Passing one, or both in agreement, is unchanged. Skipped for custom finders (the `<field>_id` value there is a lookup token, not a pk). Surfaces previously-silent contradictory input as a dev-facing failure.
* [FEAT] Add `as:` / `prefix:` to `expects` to rename the generated reader independently of the wire key. `expects :channel, as: :raw_channel` keeps `channel` as the caller-facing contract (validation, required-inputs, logging, sensitive filtering all still key off it) while exposing the value as `raw_channel` ŌĆö freeing the field's name so you can define your own method (`def channel = Channel.find(raw_channel)`). `as:` renames a single field; `prefix:` is sugar that renames several at once via literal concatenation (`expects :id, :type, on: :event_params, prefix: :event_` ŌåÆ `event_id`, `event_type`), handy for unwrapping subfields. `as:`/`prefix:` can't be combined, used with `readers: false`, or applied to a dotted `on:` path; the renamed reader clears the same reserved-name/collision checks as a field. Composes with `model:` (the record ŌĆö including the `<field>_id` lookup ŌĆö resolves against the wire key and is exposed under the alias). Subfields declared `on:` a renamed parent reference it by its reader name (the alias). Not added to `exposes` (no instance reader to free; output naming is already covered by `expose`, `expose_return_as:`, and strategy options).
* [FEAT] Add `on_enqueue_all` ŌĆö a declarative, once-per-run callback for `enqueues_each`/`enqueue_all` fan-out. It fires inside the orchestrator after the fan-out loop completes (off the clock thread), receiving the exact enqueued `count:` and an honest `sources:` hash, with flexible arity (declare a block taking neither, either, or both). The block is error-isolated ŌĆö a raising callback can never abort the fan-out. Removes the hand-rolled parent/child wrapper action previously needed for batch-level summary work (e.g. posting one Slack summary around a per-record fan-out).
* [BREAKING] A nested `call!` failure now re-raises the inner action's original exception, identical to a top-level `call!` (previously: a fresh `Axn::Failure` wrapping the inner's `result.error` with a `source:` pointer and `cause:`). `Axn::Failure` now means exactly "`fail!` was called" everywhere. `Axn::Failure#source` is removed. To reshape a child's error with context, run the child with non-bang `call` and `fail!("context: #{result.error}") unless result.ok?`. An unhandled exception that propagates through nested `call!`s is now reported to `Axn.config.on_exception` **once** ŌĆö at the innermost executor that classifies it as a reportable exception ŌĆö rather than once per nesting level (each action's own `on_exception` callback still fires at its level). A `fails_on` classification is now **sticky** to the exception object: an exception an inner action reclassified as an expected failure stays a failure (fires `on_failure`, never reported, `result.outcome` is `failure`) even when bubbled up via `call!` to an ancestor that doesn't share that `fails_on` ŌĆö so `fails_on` suppresses the report on the `call!` path too, not just `call`. The classification is per-object, so an unrelated same-class exception raised elsewhere remains a reportable exception.
* [BREAKING] Removed `error from:` and the per-message `prefix:` option on `error`/`success`. Passing `from:`/`prefix:` now raises at declaration with the migration hint.
* [INTERNAL] `step` no longer uses `error from:` / the nested `call!` wrapping; it runs each child via `call` and `fail!`s with the step's `error_prefix:` (default `"<name>: "`). A parent orchestrator that declares a base `error` now has it cascade into step failures.
* [BREAKING] A step that raises an **unexpected exception** now settles its parent as an **exception** (firing `on_exception`), not a failure. Previously the generated orchestrator blanket-`fail!`d on any step that stopped, so a bug in a step incorrectly fired the parent's `on_failure` and surfaced as `"<base>: <step>: Something went wrong"`. Now the step's outcome *category* propagates: a deliberate `fail!` (or a `fails_on`-classified exception) settles the parent as a **failure** with the composed message (unchanged), while an unclassified exception is re-raised so the parent settles as an **exception** ŌĆö `on_exception`/`on_error` fire (not `on_failure`), the global `on_exception` report still fires exactly once at the step, and the parent's `result.error` is its declared base `error` / `"Something went wrong"` (the step-name prefix is **not** applied on the exception path; the full detail, including which step raised, goes to the report). `on_error` remains the catch-all at every level. Breaking only for code asserting on a step *exception*'s parent outcome/message; step *failure* messages are unchanged.
* [FEAT] `step` accepts `if:` / `unless:` for conditional execution ŌĆö a Proc (instance-`exec`'d on the parent) or a Symbol naming a parent method, combinable (the step runs only if `if:` is truthy **and** `unless:` is falsey), evaluated immediately before the step would run. A skipped step exposes nothing and cannot fail; later steps still run. Conditions read data the same way the action does ŌĆö `expects` inputs (the reader or `inputs`) and a prior step's output via `result.<field>` (the parent must declare the exposure); a bare reference to an undeclared name raises `NameError`. A non-Symbol/non-callable condition raises at declaration.
* [BREAKING] Declaring steps **and** defining a custom `#call` on the same class now raises `ArgumentError` at load time, in either declaration order ŌĆö the generated `#call` *is* the step orchestrator, so a user-defined one collided with it (previously a silent, declaration-order-dependent override that could stop the steps from running). Use `before`/`after` hooks for setup/teardown around the steps. Subclasses adding steps to a steps-using parent are unaffected.
* [FEAT] A declared base `error "ŌĆ”"` now attaches the action's specific failure *reasons* under it ŌĆö conditional `error ŌĆ” if:`/`unless:`, entries explicitly marked `standalone: false`, and `fail!` messages ŌĆö rendered as `"<base><join><reason>"`. **Conditionality sets an entry's role, not whether it's a literal or a block:** an *unconditional* `error`/`success` (string **or** block ŌĆö they behave identically) is the **base headline**; a *conditional* one is a reason attached under the base. `standalone:` is the explicit override ŌĆö `standalone: false` attaches an unconditional entry to the base as a reason (e.g. `error(standalone: false, &:message)` for an always-on detail under the base); `standalone: true` makes a conditional reason render on its own, without the base. Default follows conditionality: an unconditional entry is standalone (headline) by default; a conditional entry is attached (reason) by default. Base/reason attachment is **gated by a declared base** (no base ŌćÆ reasons render standalone). The baseŌåöreason combination is controlled by `join:` on the base ŌĆö either a `String` separator (default `": "`) or a `->(base, reason) {}` Proc for full control such as wrapping or recasing (raise-safe: a Proc that raises or returns a non-String falls back to the default join); `join:` on a reason raises at declaration. When multiple unconditional headlines are declared, the most-recent wins (a headline whose block raises/blanks falls back to an earlier one). `success`/`done!` mirror all of this. Note: `result.error` returns the fully composed presentation string, and for **Axn-owned** failures (`fail!` and user-facing validation) the raised exception's `#message` is stamped to match it ŌĆö so rescuing from `call!` sees the same composed string; only **foreign** `fails_on`-reclassified exceptions keep their original (technical) `#message`. **Behavior change for existing actions:** an action pairing an unconditional base with a *conditional* entry will see the conditional message attached under the base by default (opt out with `standalone: true`); an *unconditional* block is now a headline (replacing, not attached) unless you add `standalone: false`.
* [FEAT] `success`/`done!` now support the same base/reason semantics as `error`/`fail!`. Conditionality sets the role exactly as for `error`: an **unconditional** `success` (string **or** block) is the base headline, while a **conditional** (`if:`/`unless:`) or explicitly `standalone: false` entry is a reason attached under the base, rendered as `"Headline: reason"`. `done!("message")` is attached under the base by default; `done!("message", standalone: true)` opts out. When no base is declared the behavior is unchanged.
* [FEAT] Add `use :model` strategy ŌĆö the ActiveRecord sibling of `use :form`. Auto-declares the contract (`expects :params, type: :params` ŌĆö accepting a plain Hash or `ActionController::Parameters` ŌĆö plus the `model: true` field, respecting a pre-declared one), exposes the record (under the field name, or `result.model` when unnamed), and saves it in a `before` hook (`fail! unless record.save`) so `call` is free for post-save logic. Attributes come from an overridable `model_params` (defaults to `params`; must return a plain Hash or *permitted* params ŌĆö unpermitted `ActionController::Parameters` raise an actionable error rather than silently bypassing mass-assignment protection), with `inject:` sugar for merging context fields (explicit `model_params` keys win on collision). An optional `prepare_model(record)` hook gives imperative pre-save tweaks (nested associations, derived fields) a home ŌĆö it runs after attribute assignment and always before the gated save. Ships mode-aware default messages ("Created/Updated <Model>" + the model's validation errors); declare a base `error "ŌĆ”"` after `use :model` to attach to the validation body, and any other override (custom `success`/`error`/`fails_on`) is a normal DSL declaration after `use :model`. Wires `fails_on ActiveRecord::RecordInvalid` as a safety net; does not auto-wrap `use :transaction` (compose it explicitly when needed). Requires ActiveRecord (raises `NotImplementedError` at declaration if it isn't loaded, mirroring `use :transaction`).
* [FEAT] Add `fails_on` declaration to reclassify chosen exception classes from the **exception** outcome into the **failure** outcome. A matching raised exception now settles as a failed result ŌĆö firing `on_failure` (not `on_exception`) and skipping the global `Axn.config.on_exception` report (on both sync and async/discarded-job paths) ŌĆö **without** wrapping it in `Axn::Failure`, so `result.exception` keeps the original and the existing `error` message resolution is unchanged. Rides `fail!`/`error` muscle memory: `fails_on ActiveRecord::RecordInvalid, "Unable to submit"`, a block receiving the exception, or an array of classes. Gives routine validation failures a declarative path out of the global handler without a manual `rescue ŌĆ” fail!`. Documented as the supported pattern for suppressing spurious `on_exception` reports when an inner action raises an expected error that an outer action handles ŌĆö declare `fails_on` on the inner action (the one that knows the exception is expected), not on the outer.
* [INTERNAL] Hardened the nested-error machinery: the per-call-tree report-dedup / `fails_on`-stickiness sets now key on exception **identity** (`Set#compare_by_identity`) rather than `object_id`, removing a latent freed-then-reused-`object_id` collision hazard; and the message resolver invokes the winning message block exactly once per resolution (previously twice ŌĆö once to probe, once to capture). `Result#error`/`#success` now memoize the resolved string, so a user-supplied message block runs a single time across the execution log and every caller read rather than once per read. `MessageDescriptor.build` (the direct/Factory path) now enforces the same `standalone:`/`join:` validation as the `error`/`success` DSL ŌĆö e.g. `join:` on a conditional reason raises at declaration instead of being silently ignored ŌĆö with that validation/defaulting consolidated in `build` rather than duplicated in the DSL layer. Also documented the `fails_on` sharp edge: classification flows outward only, so an ancestor's `fails_on` cannot un-send a report an inner action already emitted ŌĆö declare it on the action that raises the exception.
* [INTERNAL] Namespaced the framework's internal instance variables on the action instance (`@result` ŌåÆ `@__result`, `@internal_context` ŌåÆ `@__internal_context`) and the class-level `@inspection_filter` ŌåÆ `@__inspection_filter`, so a user assigning their own `@result`/`@internal_context` inside an action can no longer clobber exposed-value extraction or message rendering.
