# Changelog

## Unreleased
* [FEAT] `as:` now gives a dotted subfield key a reader (PRO-2896). A dotted subfield NAME (`expects "items.count", on: :order`) generates no reader on its own ŌĆö a dotted method name is unspeakable ŌĆö so it was purely a validation assertion; `as:` supplies a valid alias whose reader resolves the nested path segment-by-segment (`as: :item_count` ŌåÆ an `item_count` reader reading `order[:items][:count]`). This extends to `model:`: a dotted-name `model:` subfield is now usable with `as:` (`expects "order.widget", on: :payload, model: Widget, as: :widget` resolves the record from `order[:widget_id]` and exposes `widget`/`widget_id`); without `as:` it still raises (no reader to run the idŌåÆrecord lookup), now pointing at both the `as:` and dotted-`on:` spellings. A defaulted model `klass` derives from the leaf segment (`order.widget` ŌåÆ `Widget`), and a later `on:` can anchor on an aliased-dotted reader. Every "does this config generate a reader" decision now routes through one predicate (`FieldConfig#generates_reader?`, keyed on the reader name, not the wire key).
* [BREAKING] A dotted field NAME is now rejected at declaration outside a subfield (PRO-2896). `expects "a.b"` (no `on:`) and `exposes "a.b"` previously succeeded silently, defining an unspeakable, never-matching reader that failed confusingly at runtime; they now raise, pointing at `on:` (a dotted name is only meaningful as a subfield, where it denotes a nested path ŌĆö a top-level/outbound reader reads a literal key). `as:`/`prefix:` on such a key are covered by the same doctrine: `prefix:` on a dotted subfield key raises (the prefixed name stays dotted and names no reader), as does a dotted `as:` alias.
* [BREAKING] `model:` field resolution now reads keys through the canonical `Extract` path instead of a raw `[]` (PRO-2896). This unifies model reads with every other reader: a model field/subfield reading off a record or object source now dispatches the reader METHOD (matching how plain subfields already read records) rather than `[]`, and reads off a Hash are indifferent to string/symbol keys. Both align model resolution with the rest of the contract read path; a malformed source still reads as absent. Relatedly, `Extract` now reads Hash/`HashWithIndifferentAccess` sources by a direct dual-key lookup (symbol-first) instead of deep-copying via `#with_indifferent_access` on every segment, removing a per-read cost on the hot path.
* [BUGFIX] A sibling `<field>_id` default no longer manufactures a model-consistency mismatch (PRO-2889). When a `model:` field's record is already present in the wire data, axn skips writing the companion `<field>_id` default (a lookup token only needed when the record is absent) ŌĆö previously the injected id was compared against the present record's id and the call failed with a "provided record conflicts with `<field>_id`" mismatch of axn's own making. Mirrors "a present value is never overridden"; works at depth and top level.
* [BUGFIX] A dotted subfield path crossing a nested Array now resolves the same as the reader spelling (PRO-2886). `Axn::Core::FieldResolvers::Extract` dug the whole dotted path off the top-level source, checking only the immediate source's type ŌĆö so `expects "items.count", on: :payload` pushed a String key into the nested Array (`Array#dig("count")` ŌåÆ `TypeError`, which PRO-2883's guard then turned into a silent absent value), while the equivalent reader spelling (`expects :count, on: :items`) correctly reached `Array#count`. Extraction now resolves a dotted path one segment at a time, re-dispatching on the type reached at each step (Hash segment read by key, Array/object segment via its reader method), so `"items.count"` and `:count on :items` behave identically. A missing intermediate still reads as absent (PRO-2857), a Hash key colliding with an Array method name still reads the key (per-segment dispatch), and an Array reader that can't be invoked bare because it needs arguments (`items.fetch`, `items.at`) raises the typed `UnextractableError` (read as absent) rather than leaking a raw `ArgumentError` ŌĆö keeping the whole malformed-input doctrine intact on both the dotted and single-segment reader paths.
* [BREAKING] A subfield whose resolution path provably cannot be answered by any contract-valid input ŌĆö e.g. `expects "bar.baz", on: :payload` where the payload shape declares `field :bar, type: String` (and `String` has no `#baz`) ŌĆö now raises `ArgumentError` at class definition (PRO-2889, family 2). Previously the class loaded and the subfield silently read as absent on every call. Reader-style scalar access (`:count` on an `Array`, `"bar.length"` on a `String` member) remains legal (dotted-name and reader spellings resolve identically per PRO-2886's per-segment extraction).
* [BREAKING] A statically nil-tolerant field (`allow_nil:`/`optional:`/`allow_blank:`/`presence: false`) whose omission can never validate ŌĆö an unrescued required subfield anywhere beneath it ŌĆö now raises `ArgumentError` at class definition (PRO-2889, families 1+3). Previously the class loaded and every nil/omitted call failed on the buried descendant (the tolerance was a lie). The error names both declarations and the fixes; rescues (defaults at any depth, record-supplying model defaults, defaulted `<field>_id` siblings declared first) keep their contracts legal.
* [BREAKING] Value-level subfield defaults (PRO-2889): a subfield `default:` now guarantees the *resolved* value ŌĆö reader and validation both ŌĆö even when the wire write-back cannot apply it (a `model:`/non-object parent, an id-resolved or caller-supplied record whose attribute is nil, a malformed parent). Previously the same declaration was silently dead on those paths and the call failed the subfield's own validation. Relatedly, subfield validation now reads every subfield through the action's PUBLIC reader (not the raw wire value), so a user-overridden subfield reader is now exactly what validation sees. A `model:` subfield's resolution now also consults a sibling `<field>_id` subfield's value-level default when the wire carries no id token, so the record resolves from the id the caller reads (a present-but-failed id is never overridden).
* [BREAKING] A subfield `default:` no longer writes into a caller-supplied non-Hash parent (e.g. setting an attribute on a passed-in record). Previously `expects :name, on: :company, default: "x"` mutated the caller's record when its attribute was nil; now the record is untouched and the default is visible through the axn's reader/validation only (value-level defaults, see above). Read the axn's reader instead of the mutated object.
* [FIX] `input_schema` now mirrors value-level subfield defaults under a `model:` parent (PRO-2889): a nil-tolerant model whose descendants are all defaulted (usable literal default) or optional no longer over-requires the generated `<field>_id`. The old reflection assumed no default under a model could ever apply (the write-chain never synthesized a model node), so it kept the id required; value-level defaults apply at read time at any depth, so a literal-defaulted descendant is self-rescuing and the id is omittable. A Proc-defaulted descendant still keeps the id required (its value is unknowable to reflection ŌĆö the safe, stricter-than-runtime direction), as does a descendant with no rescuing signal.
* [FEAT] Added `with_ambient_context(**attrs)` test helper to `Axn::Testing::SpecHelpers` (auto-included into RSpec like `build_axn`) (PRO-2892). Block-scoped injection for `on: :ambient_context` inputs: it swaps `Axn.config.ambient_context_provider` for the block and restores it in an `ensure` (safe under nesting and when the block raises). Unlike a per-call `ambient_context:` kwarg, the provider feeds the whole call chain ŌĆö the action under test **and** any nested `.call!`/`.call`s ŌĆö so a parentŌåÆchild chain reading ambient inputs sees the injected values. It never mutates `Current` / any `CurrentAttributes`, and an explicit `ambient_context:` kwarg on a specific call still wins over the injected values. Test-only; no production behavior change. Docs also note that stubbing a `Current` reader (`allow(Current).to receive(:x)`) does **not** feed ambient inputs (the source reads the attribute hash, not the reader), and that the helper swaps a process-global provider ŌĆö isolated under process-based parallel runners (e.g. `parallel_tests`) but not thread-based ones.
* [FEAT] `SubfieldTree` is now the canonical resolved-subfield structure (PRO-2883): declaration, reflection, and the runtime `.call` path all consume one per-class artifact ŌĆö the tree plus a per-config resolved-path index (wire path + ancestor chain, aliases and dotted segments translated once at build) ŌĆö cached on the class with identity-keyed validity (copy-on-write config arrays make any declaration mint new arrays, so there are no invalidation hooks to forget and no nil-memoization footgun), deep-frozen before publication. `input_schema` stops building the tree twice per call, and subfield validation resolves each distinct `on:` chain once per call.
* [FEAT] Subfield kwarg parity (PRO-2883): `expects ..., on:` now supports the full top-level kwarg surface. `default:`/`preprocess:`/`sensitive:` work on **nested** parents (dotted `on:` paths and subfield-anchored chains) with coordinated top-down materialization ŌĆö shared intermediates materialize once, and a default is skipped when an absent ancestor's declared type can't hold an object. `coerce:` works on subfields (and `coerce_input_types` now reaches them), shape blocks (`do ŌĆ” end`) work under `on:`, and `user_facing:` works at any depth: classification follows each failing config's own declaration, any dev-facing violation still dominates, and checks whose ancestor already failed are causally suppressed (a stranded descendant no longer pages over its parent's user-facing message). Ambient subfields keep their restrictions (`default:`/`preprocess:`/`coerce:`/`user_facing:` rejected ŌĆö ambient values are framework-supplied and never read from provided_data).
* [BREAKING] Subfield `default: false` (any non-nil falsey default) now **applies** at runtime and reflects in `input_schema` (emitted, requiredness-relaxing), matching top-level key-absence semantics; previously a falsey subfield default was silently never applied.
* [BREAKING] `readers: false` is removed ŌĆö every declared field generates a reader; declaring it raises with a pointer at `as:`/`prefix:` (the rename escape hatch keeps colliding sub-keys accessible instead of suppressing them). Nothing internal or in downstream consumers used it.
* [BREAKING] Inbound validation now collects-then-settles: every contract violation ŌĆö top-level fields, subfields, and model-consistency mismatches ŌĆö is gathered before one settled exception raises, so a failing call reports ALL violations at once (previously a dev-facing subfield failure raised alone at the first violation, and a dev-facing top-level failure skipped subfield checks entirely). Causal suppression is applied post-hoc with complete failure knowledge, so a stranded descendant is pruned regardless of declaration order. Failing calls run every declared validator (no first-failure short-circuit); the happy path is unchanged.
* [BREAKING] `Axn::Internal::FieldConfig.optional?(config)` is removed ŌĆö optionality is an instance method on the config objects themselves: `config.optional?` works on both `FieldConfig` and nested `ShapeConfig` members (axn-mcp interface change; the downstream contract spec now pledges the instance method).
* [FEAT] Stranded-path diagnostics: when a nested subfield fails because an intermediate ancestor is nil (`expects :city, on: "payload.address"` with `payload.address` absent), the validation report names the stranding hop ŌĆö `'payload.address' is nil, so nested expectations beneath it cannot be satisfied` ŌĆö once per stranded chain. A nil top-level parent adds no diagnostic (its absence is self-evident in the report).
* [BREAKING] Canonical `on:` resolution: a subfield's parent now resolves through the DEEPEST reader-bearing ancestor on its chain (then raw digs for implicit segments), so both spellings of the same wire path ŌĆö `on: :b` and `on: "a.b"` ŌĆö resolve identically. For a chain crossing a `model:` subfield this means the dotted spelling now sees the resolved record (previously the raw hash) ŌĆö a corner reflection already dropped-and-warned as having no JSON representation.
* [BREAKING] One validation collector at every level: `Axn::Validation::Subfields` is merged into `Axn::Validation::Fields` (one per-config one-off collector; a top-level field validates against the inbound facade, a subfield against its canonically-resolved parent, outbound against the result facade). The settle layer is now uniform ŌĆö classification, suppression, and user-facing message composition run one per-config path with no top-level special-casing.
* [BREAKING] The model:-batch-names-its-own-`<field>_id` contradiction now raises at declaration for TOP-LEVEL `expects`/`exposes` batches too (previously only subfield batches were guarded; the top-level spelling was silently broken).
* [FIX] A malformed subfield parent (a value that can hold neither key nor method, e.g. `payload: []` where a Hash was declared) no longer crashes with a bare `Unclear how to extract` RuntimeError ŌĆö from subfield coercion, preprocess, defaults, or validation itself. Extraction raises a typed `Axn::ContractViolation::UnextractableError`, which the subfield contract machinery treats as "value absent" (PRO-2857), so the parent's own type validation classifies the bad input (and a `user_facing:` parent settles as a friendly failure).
* [FIX] A subfield declared with only presence-tolerance and no other validator (`expects :nickname, on: :user, optional: true`) no longer raises `ArgumentError: You need to supply at least one validation` on every call ŌĆö an empty validation set now means nothing to enforce, matching top-level fields. Subfields also gained the same symbol-argument delegation top-level validations have (`inclusion: { in: :some_action_method }`).
* [INTERNAL] Top-level and subfield contract machinery unified (PRO-2883): one `FieldConfig` Data type for both levels (`SubfieldConfig` removed; a top-level field is the depth-0 case), a shared one-off validator kernel (`Axn::Validation::Base`), shared parse/commit flow ŌĆö top-level `expects` now validates-before-commit like subfields, so a rescued declaration error leaves no orphaned configs or readers.
* [FIX] `Axn.config.logger` no longer returns `nil` during the Rails boot window (PRO-2891). The getter fell back to a stdout `Logger` only when `Rails` was undefined (`rescue NameError`); it did nothing when `Rails` was defined but `Rails.logger` was still `nil` ŌĆö the normal state before Rails runs its `initialize_logger` initializer. In that window `logger` returned `nil`, so any `Axn.config.logger.<level>` call site that runs at load time crashed with `NoMethodError: undefined method 'debug' for nil` ŌĆö notably `Naming.included` (naming.rb:19) when `include Axn` ran (under `Bundler.require`) on a class whose ancestor already defines `description`, breaking boot for any Rails app requiring such a gem (e.g. axn-mcp). The getter now falls back to a transient stdout logger whenever the host logger is `nil` ŌĆö not just undefined ŌĆö and, crucially, does **not** memoize that fallback, so a later call picks up `Rails.logger` once the initializer has run. Fixing the getter closes the whole class of load-time nil-logger crashes rather than guarding one call site.
* [FEAT] `coerce: :boolean` (PRO-2873 follow-up) ŌĆö adds `:boolean` to the coercible set so a `type: :boolean` field can accept a wire boolean. It coerces the case-insensitive strings `1/true/t/yes/y/on` ŌåÆ `true` and `0/false/f/no/n/off` ŌåÆ `false`, plus the integers `1`/`0`, and passes an already-boolean value through unchanged. It's the first coercer with no encoder counterpart (a boolean serializes as itself, not a string), so ŌĆö uniquely ŌĆö it also accepts a non-string wire form (the `0`/`1` integers a JSON/loosely-typed client might send for a flag); every other coercer stays string-only. Both truthy and falsy are an **explicit allowlist** (unlike Rails' "everything not falsy is true"), so an unrecognized value (`"maybe"`, `2`) is left uncoerced and fails validation rather than silently becoming `true` ŌĆö preserving coerce-or-leave. A leftover uncoerceable value (string or integer) reports "could not be coerced to a boolean" (distinct from a wrong-type value's "is not a boolean"). Works through both the per-field `coerce:` DSL and the whole-action `coerce_input_types` flag, with zero `input_schema` effect (`coerce: :boolean` reflects identically to `type: :boolean`). `BigDecimal` remains the one deferred target.
* [BREAKING] Per-class override accessors cleaned up: every setting (both config flavors) now also generates a `<name>?` predicate reading the same resolved value as a boolean, the redundant `resolved_<name>` alias is removed (it was byte-for-byte the no-arg `<name>` read ŌĆö use that), and `raw_<name>` is renamed `<name>_override` (same semantics: the stored override or the `Axn::Configurable::UNSET` sentinel, no config fallback, no callable resolution). Breaking only within unreleased pre-1.0 surface; known internal consumers (data_shifter's `raw_progress_enabled`/`resolved_*`, axn-mcp's `resolved_mcp_text_content`) migrate with the version bump.
* [FEAT] `coerce_input_types` config setting (PRO-2884) ŌĆö the operational, whole-action counterpart to the per-field `coerce:` from PRO-2873. When resolved on for an action, every top-level `expects` field whose declared type has a coercible member behaves as if it set `coerce: true`, so a fully transport-facing action (a controller handing it a params hash of strings, an adapter decoding JSON) coerces without annotating each field. It's a per-class-**overridable** setting under the `:core` namespace (via the PRO-2856/PRO-2880 machinery), giving the full lattice: global default `false` (strict ŌĆö a string where a `Date` is declared is usually a bug for an in-process Ruby caller, so we never default coercion on), a global `Axn.config.coerce_input_types = true` for an app that is entirely transport-facing (the consumer's informed choice), and a per-class `MyAction.configure { |c| c.coerce_input_types = true }` that inherits into subclasses (e.g. a controller-facing base class) without leaking to siblings or mutating `Axn.config`. A field's own `coerce:` always wins over the flag ŌĆö an explicit `coerce: false` (via the `type: { klass:, coerce: false }` form) opts one field back out of an otherwise-coercing action, and an explicit `coerce:` still works with the flag on. Non-coercible types (`String`/`Hash`/ŌĆ”) are untouched, a flag-coerced parse failure reports "could not be coerced to a Date" exactly as an explicit `coerce:` field does (while a real wrong-type non-string value still reports "is not a Date"), and `input_schema`/`output_schema` are unchanged whether the flag is on or off. Scope tracks where coercion is legal ŌĆö **top-level `expects` fields only** today (subfields are not reached); when subfield coercion lands in a future release, the flag will extend there automatically (a behavior change that will be called out then).
* [FEAT] Per-class config is now namespaced, so a single Axn composing several adapter mixins (e.g. `axn-mcp` + `axn-ruby_llm`) can carry each adapter's overrides without collision. Every Axn gains a `configure(namespace = :core) { |c| c.setting = value }` DSL (always available as `axn_configure`; `configure` defers to a base class's own method of that name if present). No-arg targets `:core` ŌĆö Axn's own overridable settings, equivalent to the flat setter. Setters are validated eagerly when the namespace's source is loaded (its `.overrides` are included ŌĆö always true for `:core`), and stored tolerantly otherwise so a library can pre-declare config for an adapter absent from the current process; a tolerant value is validated when that adapter first resolves it. Config sources declare their bucket with `config_namespace :mcp` (must precede any overridable setting) (PRO-2880).
* [INTERNAL] Schema reflection now derives each subfield tree node's requiredness/nullability in a single bottom-up pass; emission reads those annotations instead of recomputing the transitive presence obligation at each emission site. No behavior change ŌĆö this removes the class of bug where a dropped/blocked deep shape kept its runtime obligation at some sites but not others (PRO-2877).
* [BREAKING] A `model:` subfield with a dotted field NAME (`expects "org.company", on: :payload, model: X`) now raises `ArgumentError` at declaration. A dotted subfield name generates no reader, so at runtime the idŌåÆrecord lookup never runs (validation falls back to the raw Extract resolver, which digs the object, not the id) and the advertised `<leaf>_id` is unconsumable ŌĆö the contract only ever validated for a Ruby caller passing the actual nested record, never for a JSON/id client. The identical capability works under the reader spelling (`expects :company, on: "payload.org", model: X`), which the error now points at. Previously this loaded and reflection dropped+warned the config; it now fails at declaration (PRO-2877).
* [BREAKING] A `model:` subfield batch that also names its own `<field>_id` companion (`expects :company, :company_id, on: :settings, model: X`) now raises `ArgumentError` at declaration with an explanatory message. `model:` applies to EVERY field in the batch, so `:company_id` became a second `model:` subfield (it would require `:company_id_id` and reject a raw id) and collided with the raw-id reader `:company` already generates ŌĆö a normal `{settings: {company_id: 7}}` payload then failed. The `model:` subfield already exposes a `<field>_id` reader for the raw id, so the explicit id is redundant; the error says so (and it's rejected in either declaration order, since options are order-independent). Previously this raised only a bare `expects does not support duplicate sub-keys` error ŌĆö this replaces it with a message that explains the `model:`-blankets-the-batch cause. There is no working way to pair an explicit id with a `model:` subfield in one batch (declaring the id in a separate `expects` trips the same duplicate-reader guard on the generated companion) (PRO-2877).
* [BREAKING] Declaring a subfield whose `on:` names a `readers: false` parent (`expects :bar, on: :payload, readers: false` then `expects :baz, on: :bar`) now raises `ArgumentError` at declaration. A subfield parent is resolved via `public_send(parent)`, which a `readers: false` parent has no method to answer ŌĆö so the combination crashed EVERY call at runtime with `NoMethodError` (an *exception* outcome) while schema reflection still advertised the nested path with no dropped-subfield warning. The guard now consults the readers axn actually generated rather than `method_defined?`, so it also catches the case where the `readers: false` parent's name shadows an inherited public method (e.g. `:class`, `:hash`): there `public_send` silently invoked the inherited method and read the wrong object instead of raising, which is even harder to spot. Failing at declaration surfaces the mistake immediately (matching the existing fail-at-declaration guards for ambient/dotted/`user_facing:` combos); the error names the `readers: false` cause and suggests dropping it or naming a readable parent. Breaking only for a contract that could never successfully run ŌĆö nothing functioning changes (PRO-2872).
* [FEAT] Extracted the `expects:`/`exposes:` declaration-format normalizer into a public `Axn::FieldDeclarations.hydrate` (`lib/axn/field_declarations.rb`) so adapters that build Axn classes outside `Axn::Factory` ŌĆö defining `#call` themselves rather than routing a callable through the Factory ŌĆö can reuse the exact same normalization instead of duplicating it (axn-mcp's `Tool.define`, PRO-2844). It accepts an already-built Hash (passed through untouched) or an Array of bare field names (`Symbol`/`String`, hydrated to empty opts) and/or `field => opts` Hashes, returning the uniform `{ field => opts }` shape. `Axn::Factory`'s private `_hydrate_hash` now delegates to it (no behavior change); `Factory.build`'s "must provide a callable/block" contract is intentionally unchanged ŌĆö building a callable into an Axn is its whole purpose, so a no-callable "shell" mode was deliberately not added. (PRO-2878)
* [FEAT] `Axn.config` settings can now be declared `overridable: true`, giving every action class-level per-axn override accessors: `<name>(value)` sets an override, the bare `<name>` reader (and its `<name>?` boolean form) resolves the nearest override up the class ancestry and falls back to `Axn.config` when none is set, and `<name>_override` returns the override or the `Axn::Configurable::UNSET` sentinel with no fallback. Overrides live in a per-class store found by walking the superclass chain, so they inherit into subclasses, never leak to siblings, and never mutate `Axn.config`; override values are validated at set time (same `one_of:`/`validate:` as the global) and declaring `overridable: true` without an `overridable_config_source` raises at declaration. Mechanically this bridges the two config flavors ŌĆö `Axn::Configurable` (module-singleton) and `Axn::Configurable::Settings` (the class flavor `Axn.config` uses) ŌĆö onto one shared `Axn::Configurable::PerClassOverrides` generator, and the action base (`include Axn`) now includes `Axn::Configuration.overrides`. The first overridable setting is `sidekiq_job_tag_sources`: an individual action can now narrow (or disable) which declared facet types surface as its Sidekiq job tags without changing the global default, and the Sidekiq adapter resolves `sidekiq_job_tag_sources` through the override store at enqueue. (PRO-2856)
* [BUGFIX] `include Axn` no longer silently shadows a `description`/`input_schema`/`output_schema` **class** method the including class already inherits from its own (non-Axn-core) base class. Ruby places `extend`ed modules above the superclass chain in singleton method resolution, so on a class that subclasses an adapter base owning those generic names (e.g. `Axn::MCP::Tool < ::MCP::Tool`, whose `description`/`input_schema`/`output_schema` carry transport meaning) axn's own versions won ŌĆö e.g. `Tool.description("ŌĆ”")` wrote to axn's `_axn_description` and never reached the base's storage, so the value silently vanished. The `Naming`/`SchemaReflection` inclusion hooks now defer: axn layers its own `description`/`input_schema`/`output_schema` only when the name isn't already provided by a non-axn-core ancestor (the class's superclass chain, or an explicit `def self.<name>` on the class itself) ŌĆö the same discipline that gave `axn_name` its prefix. "Non-core" is scoped to `Axn::Core::*` specifically, so a satellite adapter's own DSL under a sibling `Axn::*` namespace (`Axn::MCP`, `Axn::RubyLLM`) is correctly deferred to rather than clobbered. Each deferral leaves a `debug` breadcrumb so the skipped shadow is discoverable. A plain `include Axn` (no external base owning those names) is unchanged, and a class's own explicitly-declared method already outranked `extend` and is likewise untouched. (PRO-2875)
* [FEAT] Inbound wire coercion (PRO-2873, follow-up to the PRO-2842 reflection groundwork; consumed by axn-mcp PRO-2844 / axn-ruby_llm PRO-2845): a field can opt into `coerce:` to parse an inbound wire **string** into its declared Ruby type before validation ŌĆö the inbound inverse of the outbound `Axn::Reflection::Values.serialize_value` encoder, so a JSON client sending `"2026-07-08"`/`"active"` (or a Rails form submitting `"123"`) is accepted for a `Date`/`Symbol`/`Integer` field instead of rejected for not already being the Ruby object. Sugar `coerce: <Type>` expands to a flag inside the type bag (`type: { klass: <Type>, coerce: true }`, parallel to how bare `type: Date` expands to `{ klass: Date }`); the explicit `type: { klass:, coerce: true }` form is also accepted (use it alongside sibling type options like `message:`), and unions work (`coerce: [Date, String]`, where `String` is the raw-wire passthrough branch). The v1 coercible set is exactly `Date, DateTime, Time, Symbol, Integer, Float` ŌĆö the types with a strict, unambiguous string parse (`Integer` uses base 10, so `"08"` ŌåÆ `8`); `:boolean` and `BigDecimal` are deferred to their own tickets and a target outside the set raises **not-yet-supported** at declaration. Coercion is **coerce-or-leave**: only strings are transformed (a direct Ruby caller passing a real `Date`, or a JSON-native number, is untouched; a blank string is left as-is so presence validation still fires ŌĆö a `coerce: Symbol` field can't smuggle a blank past `presence`), and an unparseable string passes through to a normal, recoverable validation error ŌĆö reported as "could not be coerced to a Date" (distinct from a wrong-type non-string value's "is not a Date"). Date/time coercion accepts only **ISO-8601-shaped** strings (a `YYYY-MM-DD` date, optionally a `T`/space-separated time with optional seconds/fraction and optional `Z`/`┬▒HH:MM` offset) before handing to lenient `.parse` ŌĆö so JSON/RFC3339, a Rails `date_field`, a no-offset `datetime-local` (read in the local zone), and Rails' `Time#to_s` all coerce, while ambiguous/partial input Ruby would otherwise guess against today's date (`"12"`, `"01/02/2026"`, a bare `14:30`) is left uncoerced and fails validation. It runs **first** in the inbound pipeline (before any `preprocess:`, defaults, and validation ŌĆö and on the async-exhaustion facet-resolution path too), is **opt-in per field** so strictness is unchanged for any field without `coerce:` and for direct Ruby callers, is scoped to **top-level `expects` fields only** (rejected at declaration on `exposes`, subfields, ambient-context subfields, and shape members ŌĆö mirroring the `preprocess:` boundary; adapters coerce deeper by walking the schema), and has **zero effect on `input_schema`/`output_schema`** (a `coerce: Date` field reflects identically to `type: Date`). The engine lives in `Axn::Reflection::Coercion` (`coerce_value` / `coercible_klasses`) as the single source of truth for the wireŌåöRuby mapping, keyed off the same class set as the encoder so the two can't drift; adapters consume the primitives rather than reinventing them. DSL misuse raises at declaration with a fix-stating message: `coerce:` + `type:` together, a boolean `coerce:` at the top level, a non-boolean explicit `coerce:` flag, a union with no coercible member, or `coerce:` outside a top-level field.

The "About to execute" before line is emitted only when success logging is on, at the success level (so an errors-only config stays quiet until something fails), and an invalid level raises `ArgumentError` at declaration. Migration: `log_calls <lvl>` ŌåÆ `auto_log <lvl>`; `log_calls false` ŌåÆ `auto_log false`; `log_calls false` + `log_errors <lvl>` ŌåÆ `auto_log <lvl>, success: false`. `Axn::Factory.build` likewise replaces its `log_calls:`/`log_errors:` keywords with a single `auto_log:` (a level/bool, or a Hash of per-outcome levels). Fixes the prior wart where setting both `log_calls` and `log_errors` silently ignored the latter.
* [BREAKING] Actions can no longer be instantiated directly ŌĆö `new` is now a private class method. This closes an unsanctioned path that bypassed hooks, inbound/outbound validation, and the rest of the per-call guarantees (callers reaching for `.new` got a half-initialized instance and were confused when none of those ran). Use `.call` / `.call!`; the rare test that needs a bare instance can use `.send(:new, ...)`. Subclasses (and the now-no-op re-inclusion case below) inherit the private `new` automatically.

Normalization is **top-level keys only**; field *values* (including nested hashes) are untouched, and symbol-*value* async behavior without ActiveJob is unchanged (still raises `Axn::Async::UnserializableArgument`). Breaking only for code that declared string field keys or relied on string-keyed direct calls failing ŌĆö both previously either failed silently or were non-idiomatic. * [FEAT] Add `user_facing:` to `expects` to mark a field whose violation is the caller's fault, not a bug. By default a failed `expects` validation is dev-facing ŌĆö it settles in the **exception** bucket, pages the global `on_exception` handler, and `result.error` is the generic `"Something went wrong"`. `expects :note, user_facing: true` reclassifies a violation of that field into the **failure** bucket instead: `on_failure` fires, the global report is skipped, and the field's validation message (`"Note can't be blank"`) surfaces on `result.error`. The value rides `fail!`/`error`/`fails_on` muscle memory ŌĆö `true` (use the field's own message), a String (override it), a Symbol naming an action method, or a callable receiving the `InboundValidationError`; one that resolves blank falls back to the field's own validation message. The surfaced message is a failure **reason**, attached under a declared base `error` by default (standalone when no base is declared), exactly like a `fail!` message. The field stays **required** (unlike `optional: true`, which removes the check) ŌĆö it changes *who is blamed*, not *whether* it's validated. It's a per-field decision: in a mixed failure where a `user_facing:` field **and** a plain field are both invalid, the dev-facing one **dominates** (the call still pages), so a real contract bug is never masked behind a friendly message. The structured `InboundValidationError` is preserved on `result.exception` either way. **`user_facing:` is top-level only** ŌĆö it can't be declared on a subfield (`on:`), and it's rejected on a field that carries nested expectations (subfields via `on:`, or a shape block); those member/structural checks and model consistency are always dev-facing (support for `user_facing:` on structured/nested fields is deferred until a concrete need appears). A matching raised exception now settles as a failed result ŌĆö firing `on_failure` (not `on_exception`) and skipping the global `Axn.config.on_exception` report (on both sync and async/discarded-job paths) ŌĆö **without** wrapping it in `Axn::Failure`, so `result.exception` keeps the original and the existing `error` message resolution is unchanged. Members accept the same options as top-level fields (`type`, `inclusion`, `optional`, `description`, ŌĆ”) and recurse via nested blocks. For arrays each element is validated with indexed errors (`element at index 2: status ŌĆ”`); for a single Hash/object value its members are validated directly. The block requires a single structured `type:` (raises `ArgumentError` on scalars, unions, or no type), composes with `of:` (which still checks element class), and ŌĆö unlike `on:` subfields ŌĆö defines **no** reader methods. Downstream tooling reads members from `config.validations[:shape][:members]`.
* [FEAT] Add `of:` array-element validation for `expects`/`exposes`. On a `type: Array` field, `of:` validates each element: a single class (`of: String`), a union (`of: [String, Numeric]` ŌĆö an element passes if it matches *any*), the `:boolean`/`:uuid`/`:params` symbols, or a `Data.define` class. Only valid alongside `type: Array` (raises `ArgumentError` otherwise, including for unions like `type: [Array, String]`). Error messages report the failing element's index (e.g. `element at index 2 is not a String`) and honor a custom `message:`. `optional`/`allow_blank`/`allow_nil` govern whether the whole field may be absent ŌĆö they do not make individual elements blank-able. Downstream tooling can read the element type from `config.validations[:of][:klass]`.
* [FEAT] `exposes`-declared fields that are also `expects`-declared are now auto-copied from the input into the result on **all** outcome paths ŌĆö success, `done!`, `fail!`, and unhandled exception. Previously, the auto-copy only ran on success/`done!` paths, leaving `result.field` as `nil` after `fail!` or an exception even when the field was provided as input. This is particularly useful for re-exposing mutated ActiveRecord objects (e.g. inspecting `user.errors` after a failed save). Explicit `expose` calls before a failure continue to work and take precedence.
* [FEAT] Execution log messages now display elapsed time in human-readable units (milliseconds, seconds, minutes, or hours) instead of always showing milliseconds
* [BREAKING] `exposes` no longer defines a direct reader method on the action instance. Exposed fields must be accessed via `result.field` (e.g., `result.greeting`). `expects` readers are unaffected. User-defined methods with the same name as an exposed field are preserved (DefaultCall still calls them). Use `result.field` in `success`/`error` message callables and `sensitive:` procs to access output values.
* [FEAT] Add boolean predicate readers for `expects` and `exposes`: `expects :enabled, type: :boolean` defines `enabled?` on the action instance; `exposes :enabled, type: :boolean` defines `result.enabled?`.
* [BUGFIX] `ExceptionContext.build` no longer raises `URI::GID::MissingModelIdError` when an exposed or expected value is an unpersisted ActiveRecord record.
